yawp
Spatial file browser. Your filesystem, dated, and seen.
Project document — overview, decisions, structure, roadmap, components, shipping.
Overview
Thesis
A day is a folder. A folder is a canvas. An agent lives inside it and makes the whole archive findable. The filesystem is the truth; yawp is the face on top of it.
Origin
Capture into dated desktop folders already works: make a folder named today, throw in every screenshot, download, note, and scrap, decide nothing. The wall is retrieval — a screenshot from three weeks ago is not keyword-searchable. yawp keeps the capture habit and solves the retrieval.
Decisions — locked
Decisions
Identity
  • This is yawp — the date-indexed canvas plus retrieval agent is the job it was missing.
Unit
  • Discrete days. One canvas per day; the calendar is the hallway, each day a room. Lightweight traversal to adjacent days; calendar stays home base.
Source of truth
  • Finder is the truth. The app is a lens over real dated folders. The Finder drop and the app drop are the same action.
  • Creating subfolders from inside the app is a requirement.
  • Dedicated root at ~/yawp, plus a one-time importer for existing dated desktop folders.
Content layer
  • Content is real files (image, .md note, .md/.webloc link, pdf, junk).
  • Furniture is canvas-native, sidecar-only: arrows, ink, frames, labels, and the agent's annotations.
  • A subfolder renders as a frame on the day's canvas in place, and double-clicks to open as its own canvas.
Calendar face
  • Hybrid weekday-style grid with heat baked into the cells (revised from a pure GitHub heatmap). Click a cell opens the day; hover previews it.
  • Chosen day view: continuous scroll — one stream of weeks, anchored on today. A month-grid view exists alongside it.
Agent
  • Embedded Claude Code in a real terminal, rooted at the yawp folder. Terminal first, pretty panel skin later. Terminal-first needs no API key.
  • Indexing and tagging run silently and automatically. Moving and reorganizing run propose-then-approve, never silent.
Retrieval
  • The moat is an agent-built index: read every file, vision-describe images, write descriptions and tags.
  • Two landings: teleport to a day and spotlight, or gather a thread across time into an ephemeral board.
Card colors
  • Six hues + neutral, each a triad (accent / fill / title / body) plus a vivid base + deep tonal. Light and dark. Hue is metadata in the sidecar. Violet reserved as identity / current.
Stack
  • Electron + Vite + React (Crisp's stack). Canvas engine: tldraw (furniture + custom FileCard shapes + store persistence).

The path
  • 01   Build — the app becomes a .app
  • 02   Identity — icon, name, bundle id, version
  • 03   Sign — a Developer ID certificate
  • 04   Notarize — Apple scans and approves it
  • 05   Package — the DMG window
  • 06   Distribute — where the download lives
  • 07   First open — Gatekeeper and quarantine
  • 08   Onboarding — the app opening for the first time
  • 09   Update — keeping it current
Steps 03 and 04 are the wall — they need an Apple Developer Program account. Everything else is free.
General pipeline
yawp
01–02 Build & identity
electron-builder turns the project into a signed-ready .app: universal binary, .icns icon, permanent reverse-DNS bundle id, entitlements.
yawp-app, Crisp stack
  • Electron + Vite + React → yawp.app
  • org.wa-de.yawp
  • file + network entitlements
03–04 Sign & notarize
A Developer ID Application cert signs the app; Apple notarizes it so Gatekeeper opens it cleanly on other Macs. Both require the $99/yr Developer Program.
Blocked — no account
  • Unsigned today; runs for you + friends via right-click → Open
  • Leave signing as the one empty field
05 Package
A DMG mounts a Finder window: background art, app icon, an Applications alias, drag to install. Configured in electron-builder's dmg block.
On-brand window
  • white bg, B&W type
  • violet arrow → Applications
06–07 Distribute & first open
Download page + GitHub Releases. Signed + notarized = one prompt then opens; unsigned = "developer cannot be verified," needs right-click → Open.
Use the existing site
  • the yawp Next.js site gets the download button
  • binary on GitHub Releases
08 Onboarding
The first-run experience: brand moment, the staged-progress card, permission priming, an honest empty state inviting the first capture.
Setup = the pitch
  • create ~/yawp → import → index + describe → connect → Ready
  • watching the agent read your archive is the demo
09 Update
electron-updater + a feed = silent background updates (also needs signing). Semantic versions, release notes, crash reporter.
Off GitHub Releases
  • updater reads the Releases feed
The account
Signing + notarization is the only hard blocker — one decision. Build the whole pipeline now as if signed; leave the signing identity as the one empty field, and "ship for real" becomes a switch, not a project.
Open — yours to design / undecided
Yours to design
  • The calendar view (continuous-scroll chosen; month grid alongside).
  • The FileCard look per type; the agent panel skin (0.3.0).
Undecided
  • Light vs dark vs both as the canvas theme (the token system makes "both" nearly free).
  • Whether all six hues are user-pickable tags or a tighter set; user-configurable root.
  • Apple Developer account: stay unsigned, personal enrollment, or borrow an account.
